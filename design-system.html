Design system.
Minimal, futuristic, calm.

The tokens and components behind the Harbor Freight project document. Monochrome, full-bleed, flat surfaces, one typeface, and Swiss scaffolding — built on the meetBean brand palette. Sage stays in reserve as an optional single accent.

Principles

Monochrome + one accent

Off-white, near-black, and gray do the work. Sage is the single accent — used only on a few deliberate moments (a dot, an active state, one node), exactly how the references use their one color. Never spread across the page.

Full-bleed, no margins

Content runs edge-to-edge with consistent edge padding — never a narrow centered column. Structure comes from rails, dividers, and alignment, not from a boxed container.

Flat + Swiss scaffolding

No gradients or shadows. Hairline rules and tracked uppercase micro-labels with bullets give every section a precise, systematic frame.

One typeface, calm type

DM Sans for everything — display and body. Contrast comes from size, weight, and spacing, not from mixing fonts. Large headings at Medium weight, generous negative space.

Rules — when to use what

The system is small on purpose. For every choice below there are exactly two answers, picked by content, never by page. If you reach for a third, you’re inventing — stop and reuse.

Rule 1 · Chips — exactly two, chosen by content

One question decides it: is it a state, or a label? Never invent a third chip.

Phase 1 In progress Roadmap

Round .pill — a live state

Sentence case, true right now. Soft shape = status. Variants .pill-solid (ink), .pill-accent (sage wash).

MUST NICE SOP

Square .tag — a fixed label

Mono uppercase, from a known set (priority, category, metadata). Systematic shape = catalog. Variants .tag-must, .tag-nice, .tag-accent, .tag-indigo, .tag-amber — each only changes the fill.

Not chips: .codechip is inline mono code (a file or field name, e.g. work_order.csv); the segmented filter control is a control. Neither is a chip.

Rule 2 · Cards — two patterns only

If it’s white + hairline + padding, it’s already .card — don’t invent a new shell.

.card — the flat info card

White surface, hairline, --r-sm radius, no shadow. The default for any block of heading + prose.

87%

.sumtile — the one metric tile

Reserved: the breakdown family (.bd-viz / .bd-cell) is strictly for “X becomes Y” transformations — not a general card.

Rule 3 · Columns — default to the equal grid

Equal peers go in a plain Tailwind grid — the one shared column pattern. A labeled checklist (mono header + rule + list) is a page-local pattern, built in the page that needs it, not a system component. Avoid bespoke pixel grid-template-columns except a genuinely asymmetric layout (a fixed sidebar beside flowing content) — and standardize even those.

Equal grid — grid md:grid-cols-2/3/4 gap-4

Peer items of equal weight — what the calm pages use.

Labeled checklist — page-local, not a system component

A mono header + hairline rule + a marked list is a useful column shape, but it was used by only one page, so it was de-promoted from the shared system. Build it locally if a page needs it — don’t add it back here.

Rule 4 · Radius & tokens

Always use the radius tokens; never hardcode 999px or 3px. Cards stay small-radius (--r-sm); only chips, pills, and buttons go --r-pill.

--r-xs · tags, codechip
--r-sm · cards, tiles
--r-pill · pills, buttons

One rule of thumb: if you’re about to write a new component class, first check whether .card, .sumtile, .pill, .tag or .bd-viz already covers it.

Rule 5 · At a glance — the one summary band

Every page opens, right after the hero, with one At a glance band — a row of 3–4 .sumtiles, each a .stat figure + a one-line label, in a grid grid-cols-2 lg:grid-cols-4 gap-4. The band gets its own section; its background follows the page’s --paper/--surface alternation so it always contrasts the section below. Same component, same place, every time — never numbers baked into the hero. Use it only with real, headline facts; a tile can hold a figure or a key term (“Phase 1”). If a page has nothing genuinely summary-worthy, skip it — a padded glance band fights calm harder than none does.

Page chrome

The things every page shares verbatim — the hero, nav, footer, motion, and type. All live in assets/css/system.css + assets/js/reveal.js so a new page inherits them by linking those two files. Never re-implement them per page.

Navigation — one canonical bar

/ Project
CurrentLive linkNot yet live

Markup is identical on every page: <nav class="site-nav"> → .nav-inner → brand on the left, .nav-tabs on the right. Each tab is an <a> (live) except the current page, which becomes <span class="cur">, and any not-yet-live page, which is <span class="dis">. Same item set, same order, on all pages — only the .cur moves.

Hero & footer — every page’s open and close

Both are one class now, not copied per page. <section id="hero" class="hero"> is the full-bleed intro — relative + clipped, content vertically centered, a hairline base, one uniform 80vh height. Inside it sits a .grid-bg mask, the optional vertical .hero-rail (the left page-number rail, lg+ only), then eyebrow + h1 + intro. The page closes with <footer class="site-footer"> — the one dark --ink bar, logo left, mono stamp right. Don’t re-declare either inline; both live in system.css.

Motion — scroll reveal

One convention site-wide. Add class="reveal" to anything that should fade-and-rise into view; assets/js/reveal.js (loaded last on every page) adds .in as it enters the viewport. No per-page IntersectionObserver — reveal.js also drives the nav’s scroll shadow.

  • .reveal · start state (hidden, +24px)
  • .reveal.in · settled (added by reveal.js)
  • load reveal.js after the page’s own scripts
  • closing.js · the one other motion script — types the closing .st-head

Type — two faces only

DM Sans for everything visible — display and body, contrast by size/weight — and JetBrains Mono for micro-labels (.mono-label). That’s the whole type system; no third typeface (Inter was removed). Load only those two in the Google Fonts link, and keep both Tailwind sans and display mapped to DM Sans.

Accessibility

The sage greens are accent colors, not text colors. On white, #70A593 and even #508875 do not meet WCAG AA for normal body text.

  • → Use sage for fills, dots, edges, large display highlights, and icon strokes.
  • → For text, use Ink (#1E1D1D) and Graphite (#5F626C) — both pass AA on Paper/Surface.
  • → White text on Sage Dark (#508875) is acceptable for large labels and buttons only.
